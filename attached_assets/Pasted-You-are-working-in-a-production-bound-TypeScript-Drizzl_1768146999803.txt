GOAL (Next Best Fix):
Standardize soft delete across the database and codebase to a single canonical mechanism:
  ✅ Canonical soft delete = deletedAt TIMESTAMP NULL
Meaning:
  - Active row: deletedAt IS NULL
  - Soft-deleted row: deletedAt IS NOT NULL (timestamp of deletion)

CURRENT PROBLEM:
Soft delete is inconsistent:
  - Some tables use deletedAt
  - Others use isActive
This causes inconsistent list queries, reporting, and “ghost record” bugs.

HARD RULES
- Do NOT break existing APIs or UI in this pass.
- Do NOT remove columns yet (isActive must remain during transition).
- Prefer additive migrations only.
- Must be reversible.
- All new/updated queries should consistently exclude deleted rows.
- Keep behavior backward-compatible during a transition window.

SCOPE
Tables currently using deletedAt only (already OK): users, customerCompanies, clientLocations, items, equipment
Tables currently using isActive only (need standardization): jobParts, locationEquipment, jobs, invoices, jobTemplates, parts (or any remaining legacy tables)
